### 2026-05-31: M7-A â€” Typed Error Hierarchy for applyFeedback / applyFeedbackById (Edgar)

**Author:** Edgar (Learning Systems Specialist)  
**Date:** 2026-05-31  
**Branch:** `eureka/m7-a-typed-errors`  
**Status:** SHIPPED (PR #38 opened)

**Decision:** Introduce a typed error class hierarchy in `packages/eureka/src/activities/errors.ts`, replacing all six generic `throw new Error/TypeError/RangeError(...)` sites in `applyFeedback` and `applyFeedbackById` with domain-specific typed subclasses.

**Error classes introduced:**
- `FactNotFoundError` (extends `Error`) â€” FactReader returns `null`
- `InvalidFeedbackOptionsError` (extends `Error`) â€” `correctionDelta` undefined for `user_correction`
- `InvalidTrustValueError` (extends `RangeError`) â€” value non-finite/out-of-range
- `FactReaderContractError` (extends `TypeError`) â€” FactReader returns `undefined`
- `UnhandledFeedbackEventError` (extends `TypeError`) â€” exhaustive `switch` `never` branch

**Discriminator pattern:** Every class carries `readonly code: '<CODE>'` for narrowing without `instanceof`.

**Canonical narrowing policy (M7-A Cycle 1):** Use `err.code === '...'` as the **primary** discriminator. `instanceof` is convenience-only â€” it can fail across ESM realms. `code` is realm-safe. M7-B narrowing tests will exercise `code` exclusively.

**Rationale:** (1) Caller narrowing â€” generic throws are indistinguishable. (2) Zero behavior change â€” all 40 existing tests pass without modification. (3) M7-B prep â€” `code` discriminators are the primary hook for exhaustive narrowing. (4) Message preservation. (5) `Object.setPrototypeOf` defensive call in constructors.
