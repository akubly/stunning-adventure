### 2. Ancestry Integration in Knowledge Graph

**Outcome:** Prescription Ancestry as Graph Patterns

**Problem:** Prescription ancestry stored as linear JSON chain. Difficult to query "what if" branches or detect cycles in ancestor relationships.

**Solution:** Dual representation
- **Linear (L1):** Prescription ancestry JSON for fast serialization and export
- **Graph (L2):** Edges table with ancestry links. Node = decision or prescription. Edge = "ancestry" (who decided to apply me?) or "drift" (how much did I change the outcome?).

**Ancestry Edge Types:**
- `ancestry` (decision → prescription)
- `caused_drift` (prescription → outcome change, quantified)
- `feeds_into` (decision → downstream decision)
- `refutes` (outcome contradicts ancestor prediction)

**Example Query:** Find all prescriptions in ancestry chain whose application caused >10% outcome drift:
```sql
WITH RECURSIVE ancestry AS (
  SELECT source_id, target_id, weight FROM knowledge_graph_edges
  WHERE relation_type = 'caused_drift' AND weight > 0.1
  UNION ALL
  SELECT a.source_id, kg.target_id, a.weight * kg.weight
  FROM ancestry a
  JOIN knowledge_graph_edges kg ON a.target_id = kg.source_id
  WHERE relation_type = 'caused_drift'
)
SELECT DISTINCT source_id FROM ancestry;
```

### 3. Caching Layers for Wiki & Knowledge Graph

**Outcome:** Multi-Layer Cache Strategy for Graph Queries

**Problem:** Recursive CTE queries on large graphs are expensive. Need caching strategy.

**Solution:**
- **L1 (In-Memory):** Node payload cache (title, description, links). ~10ms refresh on update.
- **L2 (Session Store):** Query result cache (ancestry chains, drift detection). ~5 min TTL.
- **L3 (Short-TTL):** Traversal index cache (precomputed BFS from common root nodes). ~1 hour TTL.
- **L4 (Long-TTL):** Archived graph snapshots (full graph state at decision points). ~30 day TTL.

**Cache Invalidation:** When new decision added or prescription applied:
1. Invalidate L1 nodes touching new decision (ancestors + descendants)
2. Invalidate L2 query results mentioning affected nodes
3. Invalidate L3 traversal indices if root node affected
4. Append new graph state to L4 archive

**Materialized Views:** For common queries (top 10 highest-impact decisions, all prescriptions by user, convergence patterns):
- Pre-compute at L3 refresh cycle (~1 hour)
- Store in separate table with versioning
- Update async (non-blocking to user queries)

### 4. Wild Cards: Time-Travel Debugging & Predictive Cache Warming

**Outcome:** Two Wild Cards Scoped for Phase 6+ Backlog

**Time-Travel Debugging:**
- User selects any decision point in ancestry chain
- Rewind session state to that point
- Replay with different model, tool, or parameter choice
- Observe outcome delta compared to original choice
- Use case: "What if I'd used gpt-4 instead of gpt-3.5?" → measure quality + cost difference

**Implementation:** Store session snapshots at decision points in L4 cache. Replay uses same tool invocations but swaps model/tool. Requires deterministic tool output (difficult; may need mock layer for testing).

**Predictive Cache Warming:**
- User starts new session similar to past sessions (same user type, similar model preferences)
- Forecast likely tool invocations based on user history
- Pre-populate L2-L3 cache with anticipated results
- Reduces latency on first tool call in new session

**Implementation:** ML-based predictor (user_profile → expected_tools). Train on historical session data. Phase 5 implementation (post-canary metrics available).
